## Роль и контекст

Ты — старший backend/ML-инженер. Нужно реализовать с нуля прототип семантической поисковой системы для русскоязычных текстов — программную часть выпускной квалификационной работы (ВКР) на тему «Разработка семантической поисковой системы для русскоязычных текстов». Архитектура, функциональные и нефункциональные требования уже зафиксированы в техническом задании (приведено ниже целиком) — не меняй их произвольно, реализуй именно их. Там, где требования оставляют выбор (конкретная версия модели, конкретная библиотека BM25 и т. п.), выбирай конкретную реализацию сам и явно документируй выбор в README и в комментариях кода.

Это учебный, но полноценно работающий прототип: код должен реально запускаться и искать по коллекции документов, а не быть заглушкой. Метрики качества, упомянутые в задании (NDCG@10 65,71–65,85 у гибридных конфигураций против 52,16 у BM25 на бенчмарке RusBeIR), — это ориентиры из независимого исследования (глава 2 ВКР), а не результаты, которые нужно воспроизвести тестами; в тестах проверяй только относительный порядок (гибрид не хуже базовых режимов на тестовом наборе), а не конкретные числа.

## Технологическое ограничение

Используй только технологии, зафиксированные в разделе 7 требований (см. ниже): Python, FastAPI, `transformers` + `sentence-transformers`, BM25-реализация, векторное хранилище с поддержкой фильтрации по метаданным, Docker/Docker Compose. ## ТЕХНИЧЕСКОЕ ЗАДАНИЕ (полный текст требований — источник истины)

### 1. Общие сведения

**1.1 Наименование.** Семантическая поисковая система для русскоязычных текстов (СПС).

**1.2 Назначение.** Полнотекстовый поиск по коллекциям документов на русском языке с использованием лексического (точное совпадение слов), семантического (смысловое сходство) и гибридного поиска.

**1.4 Область применения.** Русскоязычные коллекции документов произвольного домена (юридические, научно-технические, вопросно-ответные и иные), где важно одновременно точное совпадение терминов/кодов и смысловое сопоставление.

### 2. Термины

- **Лексический (разреженный) поиск** — точное совпадение слов запроса и документа (алгоритм BM25).
- **Семантический (dense) поиск** — сопоставление векторных представлений (эмбеддингов) в общем пространстве.
- **Гибридный поиск** — объединение результатов лексического и семантического поиска.
- **Чанкинг** — разбиение длинного документа на фрагменты перед индексацией из-за ограничения длины входа энкодера.
- **Векторное хранилище** — БД, оптимизированная для хранения векторов и приближённого поиска ближайших соседей (ANN).
- **Переранжирование (reranking)** — повторное упорядочивание топ-N кандидатов более точной моделью совместной обработки пары «запрос — документ».

### 3. Цели системы

1. Качество поиска гибридной схемы должно быть не хуже, чем у чисто лексического или чисто семантического подхода по отдельности.
2. Устойчивость к морфологическому разнообразию русского языка, лексическому разрыву запрос/документ, узкоспециализированным терминам и аббревиатурам.
3. Управляемые пользователем механизмы уточнения выдачи: обязательные/исключаемые слова, исправление опечаток.
4. Модульность и воспроизводимость для дальнейшего развития без пересмотра архитектуры.

### 4. Функциональные требования

#### 4.1 Модуль индексации (offline)

| № | Функция | Реализовать как |
|---|---------|------------------|
| Ф1.1 | Приём и предобработка документов | Загрузка txt/HTML/уже-извлечённого текста из PDF/DOCX; очистка разметки, юникод-нормализация (NFC), удаление лишних пробелов |
| Ф1.2 | Настраиваемый чанкинг | Минимум 3 стратегии: `fixed_window` (по токенам/символам с перекрытием), `sentence_window` (N предложений со скользящим окном), `paragraph` (по абзацам). Стратегия и параметры (`chunk_size`, `overlap`, `window_sentences`) задаются конфигом, не кодом |
| Ф1.3 | Векторизация фрагментов | Sentence encoder из `sentence-transformers`, батч-инференс |
| Ф1.4 | Построение лексического индекса | BM25 по тем же чанкам; опциональная лемматизация для русского (флаг в конфиге) |
| Ф1.5 | Запись в векторное хранилище | Вектор + метаданные (`source`, `doc_id`, `chunk_id`, `section`, `date`, произвольные user-поля) |
| Ф1.6 | Версионирование индекса | Каждая индексация получает `index_version` (хеш от: имя модели + версия конфига чанкинга + timestamp); хранится манифест версий, поддержан откат |

#### 4.2 Модуль обработки запроса (online)

| № | Функция | Реализовать как |
|---|---------|------------------|
| Ф2.1 | Приём запроса | HTTP POST с текстом запроса + опциональными параметрами |
| Ф2.2 | Исправление опечаток | Расстояние Левенштейна / spell-checker; возвращать предложенное исправление в ответе API, не блокируя выполнение исходного запроса |
| Ф2.3 | Разбор must-contain / must-exclude | Явные поля запроса `must_contain: list[str]`, `must_exclude: list[str]` |
| Ф2.4 | Расширение запроса словарём терминов | Простой JSON/YAML словарь «аббревиатура/термин → раскрытие/синонимы», подключаемый и редактируемый без изменения кода |
| Ф2.5 | Векторизация запроса | Тот же экземпляр модели, что при индексации (гарантия сопоставимости векторного пространства) |

#### 4.3 Модуль поиска и ранжирования

| № | Функция | Реализовать как |
|---|---------|------------------|
| Ф3.1 | Dense-поиск | ANN top-k по косинусному сходству |
| Ф3.2 | Лексический поиск (BM25) | top-k по BM25-скору |
| Ф3.3 | Гибридизация | Reciprocal Rank Fusion (RRF) как основной метод + опциональная взвешенная сумма нормализованных скоров как альтернативный режим |
| Ф3.4 | Пост-фильтрация | Строгий фильтр must-contain/must-exclude (точное совпадение слов, независимо от dense-скора) + фильтр по метаданным |
| Ф3.5 | Переранжирование | Cross-encoder по топ-N (N настраивается, дефолт 50) после гибридизации и фильтрации; включено по умолчанию, отключаемо флагом |
| Ф3.6 | Формирование ответа | Ранжированный список: `chunk_id`, `doc_id`, текст чанка, `score`, метаданные, (опционально) подсвеченные совпадающие термины |

#### 4.4 Администрирование и мониторинг

| № | Функция |
|---|---------|
| Ф4.1 | Просмотр статуса индекса, запуск переиндексации, откат к предыдущей версии |
| Ф4.2 | Структурированное логирование запросов: текст, время отклика, режим поиска, версия индекса, параметры |
| Ф4.3 | Переключение режима поиска (`dense` / `bm25` / `hybrid` / `hybrid_rerank`) через конфиг/параметр запроса, без изменения кода |

### 5. Нефункциональные требования

- **Качество.** Гибридный режим на тестовом наборе запросов не должен давать NDCG@10 хуже, чем лучший из базовых режимов по отдельности — покрыть интеграционным тестом на синтетическом наборе (см. раздел «Тестовые данные»).
- **Производительность.** ANN-поиск — через HNSW-совместимое хранилище (см. раздел 7); реранжирование — только по ограниченному топ-N, никогда по всей коллекции.
- **Модульность.** Слабая связанность модулей: препроцессинг / чанкинг / эмбеддер / vector store / лексический индекс / гибридизатор / reranker / API — так, чтобы замена эмбеддера или векторного хранилища не требовала правок в других модулях (проверяется явным интерфейсом/абстрактным классом на каждый модуль).
- **Воспроизводимость.** Версия модели, стратегия чанкинга и версия индекса — фиксируются и логируются на каждый прогон; переиндексация — не инкрементальный патч, а полная пересборка новой версии индекса.
- **Языковая поддержка.** Корректная обработка кириллицы, смешанных ru/en текстов (термины, аббревиатуры на латинице), опциональная лемматизация для лексической ветви.
- **Надёжность.** Отказ необязательного этапа (reranker, расширение запроса) не должен приводить к отказу всего запроса — деградация до базового режима с логированием предупреждения.

### 6. Архитектура (два слоя)

```
ИНДЕКСАЦИЯ (offline)                     ОБРАБОТКА ЗАПРОСА (online)
Документы → Препроцессинг → Чанкинг      Запрос → исправление опечаток
→ Эмбеддер → { Векторное хранилище,        → расширение словарём терминов
               Лексический индекс }         → разбор must-contain/must-exclude
                                            → Эмбеддер (тот же инстанс)
                                            → { Dense-ретривер, BM25-ретривер }
                                            → Гибридизация (RRF)
                                            → Строгий лексический фильтр
                                            → Переранжирование (cross-encoder)
                                            → Ранжированная выдача
```

Обязательные модули (каждый — отдельный класс/пакет за абстрактным интерфейсом):
`preprocessing`, `chunking` (стратегии как отдельные классы одного интерфейса), `embedder`, `vector_store` (интерфейс + конкретная реализация), `lexical_index` (BM25), `hybridizer` (RRF/weighted fusion), `reranker`, `query_processing` (typos, expansion, must-contain/exclude), `api` (FastAPI), `admin` (версии индекса, логирование).

### 7. Технологии

| Компонент | Технология |
|-----------|-----------|
| Язык | Python 3.11+ |
| Энкодер/reranker | `transformers`, `sentence-transformers` |
| API | FastAPI + Uvicorn |
| Лексический поиск | BM25-реализация |
| Векторное хранилище | С поддержкой фильтрации по метаданным (ANN/HNSW) |
| Контейнеризация | Docker / Docker Compose |

### 9. API-контракт (обязательные эндпоинты)

| Эндпоинт | Метод | Назначение |
|----------|-------|------------|
| `/index` | POST | Индексация документа(ов): препроцессинг → чанкинг → векторизация → запись в vector store + лексический индекс |
| `/search` | POST | Поиск: параметры `query`, `must_contain`, `must_exclude`, `mode` (`dense`/`bm25`/`hybrid`/`hybrid_rerank`), `top_k` → ранжированный список результатов |
| `/reindex` | POST | Полная переиндексация коллекции (новая версия индекса) |
| `/health` | GET | Статус сервисов: vector store, embedder, reranker, текущая версия индекса |

### 10. Критерии приёмки (реализовать как автотесты)

1. Поиск работает в режимах `dense`, `bm25`, `hybrid` и возвращает разные, но валидные ранжирования.
2. При `hybrid_rerank` итоговый порядок отличается от `hybrid` за счёт реранжирования топ-N.
3. `must_contain`/`must_exclude` корректно включают/исключают документы независимо от векторного скора.
4. Замена embedder или vector store (через конфиг/DI) не требует изменения кода API-слоя, гибридизатора, препроцессинга.
5. Повторная индексация одного корпуса с фиксированной моделью и стратегией чанкинга даёт идентичный набор чанков и векторов (детерминизм).
6. На тестовом синтетическом наборе NDCG@10 гибридного режима ≥ NDCG@10 каждого базового режима по отдельности.

## Рекомендуемые конкретные библиотеки (выбери и зафиксируй)

Эти конкретные пакеты не были жёстко предписаны заданием (см. раздел 11 «Ограничения» — версия модели выбирается на этапе реализации) — используй их как разумный дефолт, если нет других указаний:

- **Embedder:** `BAAI/bge-m3` (мультиязычная, до 8192 токенов, хорошо показала себя на RusBeIR) через `sentence-transformers`. Сделай имя модели параметром конфига, а не константой в коде.
- **Reranker:** `BAAI/bge-reranker-v2-m3` через `sentence_transformers.CrossEncoder` — та же экосистема, поддерживает русский.
- **BM25:** `rank_bm25` (простая интеграция, чистый Python) — оберни в свой интерфейс `LexicalIndex`, чтобы можно было позже заменить на встроенный полнотекстовый поиск другого хранилища.
- **Векторное хранилище:** `Qdrant` через `qdrant-client`. Для локальной разработки используй встроенный режим `QdrantClient(path="./data/qdrant")` (без Docker); для продакшена — `docker-compose` с сервисом `qdrant/qdrant`. У Qdrant есть встроенная фильтрация по payload (метаданным) — это закрывает Ф1.5/Ф3.4.
- **Лемматизация (опционально, флаг в конфиге):** `pymorphy2` для русского.
- **Опечатки:** `rapidfuzz` (расстояние Левенштейна) поверх словаря известных термов из индекса.
- **Конфиги:** `pydantic-settings` + один `config.yaml` (модель, стратегия чанкинга, веса RRF, top_k, rerank on/off, путь к словарю терминов).

Если сочтёшь другой набор более подходящим (например, `FAISS` вместо Qdrant для совсем простого прототипа без фильтрации) — обоснуй выбор в README и убедись, что абстракция `VectorStore` позволяет переключиться без переписывания остального кода.

## Тестовые данные

Программа на момент подготовки ВКР не реализована и не имеет реального корпуса. Создай **синтетический** русскоязычный тестовый корпус из 30–50 коротких документов (2–5 предложений каждый) на 3–4 условных тематики (например: юридические нормы, ИТ-термины/аббревиатуры, общие новости) с несколькими встроенными «сложными» случаями:
- пары «запрос — документ» с лексическим разрывом (синонимы без совпадения слов) — для проверки, что dense-поиск находит то, что BM25 не находит;
- документы с точными кодами/аббревиатурами — для проверки, что BM25/must-contain находит то, что dense может упустить;
- разметь для каждого тестового запроса ожидаемый релевантный документ (relevance judgments) в отдельном JSON — на основе этого считай NDCG@10 в интеграционном тесте.

**Явно пометь в README и в коде/данных, что этот корпус синтетический и создан только для демонстрации и автотестов**, а не является результатом реального сбора данных.

## План реализации (веди по фазам, после каждой — рабочий, тестируемый результат)

1. **Каркас проекта.** Структура папок, `config.yaml`, `pydantic-settings`, заготовки абстрактных интерфейсов (`Chunker`, `Embedder`, `VectorStore`, `LexicalIndex`, `Reranker`) как ABC/Protocol.
2. **Индексация (offline).** Препроцессинг → чанкинг (3 стратегии) → эмбеддинг → запись в vector store + BM25-индекс → версия индекса и манифест. Юнит-тесты на каждый шаг.
3. **API индексации.** `/index`, `/reindex`, `/health`. Проверить на синтетическом корпусе.
4. **Поиск (online), базовые режимы.** Dense-ретривер, BM25-ретривер, режимы `dense`/`bm25` в `/search`.
5. **Гибридизация и фильтрация.** RRF, must-contain/must-exclude, режим `hybrid`.
6. **Обработка запроса.** Исправление опечаток, расширение словарём терминов — как опциональные шаги перед векторизацией запроса.
7. **Переранжирование.** Cross-encoder, режим `hybrid_rerank`, деградация при отказе.
8. **Администрирование.** Логирование запросов, просмотр версий индекса, откат.
9. **Docker.** `Dockerfile` + `docker-compose.yml` (сервисы: `app`, при необходимости `qdrant`).
10. **Тесты приёмки.** Один интеграционный тест на каждый пункт раздела 10 требований, синтетический корпус с relevance judgments, расчёт NDCG@10 по режимам.
11. **README.** Как запустить локально и в Docker, описание конфигов, обоснование выбора конкретных библиотек, ограничения (соответствуют разделу 11 требований — ВКР).

## Формальные требования к коду

- Именование классов/функций/переменных — на английском (стандартная практика), комментарии и docstring — можно на русском, если это упростит защиту ВКР.
- Каждый модуль — за интерфейсом (ABC/Protocol), внедрение зависимостей через конфиг, а не хардкод конкретного класса.
- Обязательны логирование (`logging`/`structlog`) и типизация (type hints везде, `mypy`-совместимо).
- Не используй технологии за пределами раздела 7 требований без явного обоснования в README.
- Не подделывай качество: если реранжирование или расширение запроса не даёт эффекта на синтетических данных — это нормально, суть тестов — проверить работоспособность механизма, а не переиграть тестовые данные под желаемый результат.

## Definition of Done

- `docker-compose up` поднимает работающий сервис.
- Все автотесты (юнит + интеграционные) проходят.
- `/health` возвращает `ok` со статусом всех подсистем и текущей версией индекса.
- README позволяет с нуля поднять систему и прогнать демонстрационный поиск по синтетическому корпусу за < 10 минут.
